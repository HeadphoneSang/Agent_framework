# 如何确保LLM输出指定结构的JSON文本：
## 从结构上:
- 我们使用`function_call`与`function_choice`来强制模型调用谋个函数来回答问题。
- 只要是模型支持强制的`function_choice`就可以让确保输出的json结构正确。
# 但是已有的一些问题：
- 有的模型的`function_choice`是软约束，比如**deepseek**，当我在调用**deepseek**的时候，即使我给了他`function_choice`的约束，他也是会出现幻觉调用其他的函数，从而导致输出结构不符合预期。
## 解决问题的一些思路：
### 我们可以从三个层面由浅至深的解决这个问题：
- **提示词开始预防**：精细化prompt，通过精准的格式引导和少量的few shot，让模型从一开始的生成阶段就能映射到正确的语义空间。从根本上解决输出格式的问题。
- **错误检测**：对llm生成的结果进行及时的校验。一旦发现结构性的错误或者是非法的数据，立刻将错误信息打包成消息，再次发给LLM，让LLM自动修复。
- **错误兜底**：当LLM的检测超过最大的epoch后，还是未能正确修正错误。我们立刻执行兜底策略，比如回复固定的回答格式，或者将错误的function_call包装成一个固定的思考文本输出，然后继续执行function_call。确保业务的流畅性。
